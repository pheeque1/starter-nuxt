Implementation plan

fortrabbit.yml

A YAML config for apps and environments that mirrors the dashboard, works with git push, and keeps frapi as the source of truth.

git push as save dashboard drift visible downloadable platform YAML
Goal

Configuration should feel like the dashboard, just stored in git.

Exact platform shape

App config, environment config, domains, deployment settings, and components map to dashboard concepts.

Git push saves config

On push, frapi reads the config and updates the target environment when the file is valid and the actor is allowed.

Dashboard remains transparent

The dashboard shows applied state, drift, source repo, last commit, and events for every config change.

Repository layout

One directory, app file, shared base, environment files.

.fortrabbit/
  app.yml
  base.yml
  production.yml
  staging.yml
  en-123tg5.yml
Directory Use .fortrabbit/ as the main name. It is clear and works well in templates.
Base rule If base.yml sits beside an environment file, it is inherited and overridden by that file.
App config app.yml owns app-level settings such as region and app details.
Two representations

Git says what the user wants. The platform export says what exists.

Git version

Written by humans. It may use friendly filenames, skip defaults, or be behind dashboard edits.

production.yml
staging.yml

Platform version

YAML generated from frapi. It is pretty, complete, and includes IDs, defaults, and current values.

en-123tg5.yml
en-145xg5.yml

Users can download the real platform config from the dashboard at any time.

Environment identity

Environment IDs are preferred. Names are accepted aliases.

Priority Resolver Behavior
1 Explicit id inside the YAML Use it as the environment identity. If name differs, treat the name as a requested rename.
2 Filename is an existing environment ID Use the filename as the stable identity, useful across multiple repositories.
3 Explicit name inside the YAML Match by name. If more than one match is possible, stop and ask for an ID.
4 Filename stem Use it as the environment name. If no environment exists, treat it as a creation request.
YAML example

App-level config is small and stable.

# .fortrabbit/app.yml
id: ap-bucks1
name: Bucks
team: tm-park5a
region: eu-loc
software:
  preset: craft-cms
  version: "4.0"
Region Defined on the app. It is used during creation and cannot be changed after the app exists.
Team Used during new app creation. If an existing app belongs to another team, show drift and do not move the app.
Software Uses real dashboard preset slugs like craft-cms, laravel, wordpress, and general-php.
ID Optional in user-created YAML. Present in platform exports and useful when syncing existing apps.
YAML example

Base config removes repeated environment settings.

# .fortrabbit/base.yml
php:
  version: "8.4"
node:
  version: "24"
deployment:
  permission: repositoryAccess
  trigger: push
  strategy: merge
  buildCommands:
    - composer install --no-dev --prefer-dist --no-interaction --no-ansi --no-progress
    - npm install
  postDeployCommands:
    - php craft migrate/all
    - php craft project-config/apply
    - php craft clear-caches/all
  excludePatterns:
    - /node_modules
  replacePatterns:
    - /config/project
  cachedDirectories:
    - node_modules
    - vendor

Inheritance rule

Every sibling environment file inherits this file. The environment file only needs to declare what differs.

This mirrors the dashboard deployment settings: permission, trigger, strategy, build commands, post-deploy commands, excludes, replaces, and cached directories.

Environment files can also cover runtime, env vars, firewall, jobs, and crons.

  CRAFT_ENVIRONMENT: staging
  CRAFT_STREAM_LOG: "true"
  CRAFT_SECURITY_KEY:
    required: true
    secret: true
    fromPreset: craft-cms
firewall:
  incoming:
    waf: true
  outgoing:
    - description: Search API
      port: 9200
    - description: External service range
      ports: [3000, 3010]
jobs:
  - name: queue-worker
    type: worker
    command: php craft queue/listen
    isActive: true
    maxRuntime: 300
    exitTimeout: 30
    exitSignal: TERM

  - name: clear-caches
    command: php craft clear-caches/all
    expression: "0 3 * * *"
Env vars Template YAML can declare required values. The new app flow asks for the missing values before creation.
Runtime PHP and MySQL versions, update behavior, and PHP settings are environment config.
Workers Jobs and cron commands belong beside the environment that runs them.
YAML example

Environment IDs remove cross-repo ambiguity.

# .fortrabbit/en-ptjghl.yml
id: en-ptjghl
name: production
branch: staging

  trigger: manual
  strategy: replace
  excludePatterns:
    - /web/uploads
    - /storage/runtime

  php:
    plan: XS
  database:
    plan: XS
  backups:
    plan: Backups XS
  jobs:
    plan: Jobs XS

Resolution

If id is present, it wins. The filename may also be the ID. The name remains mutable config, so this file can request a rename if the actor has permission.

This is the preferred style for multiple repositories because every repo points to the same environment without guessing.

YAML cases

The resolver handles names, IDs, new environments, drift, and denied changes.

YAML case Example Result
Existing environment by ID id: en-ptjghl apply ID identifies the environment.
Existing environment by filename ID en-ptjghl.yml apply Filename identifies the environment.
Existing environment by name name: production apply Name is accepted if unique.
New environment preview-pr-42.yml create Subject to create permissions and app limits.
Ambiguous name name: staging reject Ask for an environment ID.
Immutable app region differs region: us-e1a drift Show it in the dashboard and deploy logs; do nothing.
Existing app team differs team: tm-park5a drift Show it in the dashboard and deploy logs; do not move the app.
Denied component booking jobs: { plan: Jobs XS } drift Current booking stays unchanged and deployment reports the denied update.
File removed from git production.yml deleted unmanaged Stop git-based management; do not delete the environment.
Apply model

Git push behaves like saving config in the dashboard.

01
Receive push
Detect .fortrabbit/ in the pushed tree.
02
Parse config
Merge base.yml, resolve app and environment files.
03
Validate
Run schema, identity, permission, and business rules.
04
Apply state
Update frapi state to match the requested config.
05
Report result
Write deploy logs, events, drift, and dashboard status.
Multiple repositories

The environment is the target. The last successful config push wins.

Supported

Different repositories may define environments for the same app. This enables split source ownership while keeping platform state centralized.

Tracked source

Store the winning source as repo, branch, commit, pushed actor, and timestamp. Show it in the dashboard.

Repository Environment file Result Dashboard state
shop/frontend production.yml applied Current source until another valid push targets the same environment.
shop/worker en-123tg5.yml applied later Becomes the current source because it resolved to the same environment ID.
Dashboard conflict

Dashboard edits do not write back to repositories.

Current state

frapi remains the platform state store. Dashboard saves update that state directly.

Drift

If git config differs from the platform state, show drift in the dashboard.

Export

Offer a downloadable .fortrabbit/ directory that users can commit. Later, offer to create a pull request for them.

No dashboard ignore setting for v1. Removing the file from git stops git-based management for that environment. It should not silently delete the environment.

Permissions

Component bookings are allowed only when the actor is allowed.

A developer can push a config file that requests booking changes. The platform should validate the pushing actor against the same permission model used by the dashboard.

Case Outcome
Actor has booking permission apply Update components and record the event.
Actor lacks booking permission reject Keep current booking, show drift, and write it to the deploy log.
Events

Every config change needs an audit trail.

Git-driven event

Actor, repository, branch, commit, environment, changed resources, check result, and apply result.

Dashboard-driven event

Actor, UI surface, changed resources, previous value, new value, and resulting drift if a git config exists.

This makes the dashboard answer the operational question: who or what is responsible for the current state?

Scope

The YAML should cover app source config, not account management.

Scope Covered Notes
v1 App name, team at creation, region at creation, software, environments, deployment, env vars, domains, components, firewall, PHP/MySQL settings, jobs, crons. These map directly to app or environment config.
references Secrets, SSH keys, payment method, people, team access. Use dashboard input or references. Do not store secret values or personal access config in git.
later Create a pull request from the dashboard export. First ship download. Later, let the dashboard commit the platform config back to git.
dashboard only Billing details, user settings, team settings, logs, metrics, backup restore, deploy, restart, delete. These are actions, account settings, or observability, not repository config.
Implementation sequence

Build the contract first, then the UI.

1. Schema Define app.yml, base.yml, environment YAML, name/ID lookup, inheritance, and platform export format.
2. Parser Read pushed tree, merge base config, resolve environment aliases, and produce a normalized plan.
3. Validator Run file shape, permission, fixed region, domain, component booking, and multi-repo checks.
4. Apply engine Apply accepted changes to frapi state. Source, actor, commit, and drift are stored in frapi.
5. Dashboard Show drift, events, download, deploy logs, and template review.
